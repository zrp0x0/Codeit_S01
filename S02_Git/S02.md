# Git 브랜치의 이해와 활용

---

## 1. Git의 작업 관리 개념 이해

### 브랜치를 통해 작업 
- 브랜치
    - 분기점 / 메인 코드에서 갈라져 나온 독립적인 개발 공간을 의미
    - 새로운 기능 추가, 버그 수정, 실험 등 다양한 작업을 수행할 수 있게 해줌
    - 독립적인 공간뿐만 아니라 Git의 특정 커밋을 가리키는 포인터로도 작동함
    - git branch로 추가 브랜치를 만들 수 있음 (git init => main branch)
        - TMI: Github에서는 브랜치 이름을 master => main을 권장함

- 사용하는 이유
    - 보다 안전한 개발을 위해서 사용함
    - 현업의 효율성
    - 버전 분리 및 관리: 버전 별로 만들어놓으면 롤백이 쉬워짐

### 태그를 통해 버전 시작점 표시
- 태그란?
    - 특정 커밋에 이름표를 붙여 쉽게 식별할 수 있도록 하는 기능
    - ex. git tag v1.0 -> git checkout v1.0으로 사용 가능

- 태그를 사용하는 이유
    - 중요한 버전 기록: 명확한 배포 지점 지정
    - 롤백 용이: 문제가 생기면 태그 시점으로 쉽게 되돌릴 수 있음
    - 문서화 효과: 릴리즈 노트와 연결하기 좋음

- 브랜치 vs 태그
    - 작업 공간 | 특정 시점을 고정하기 위해 사용
    - 커밋을 계속 따라감 | 고정 시점
    - 기능 개발, 버그 수정, 실험 등 | 배포, 릴리즈, 마일스톤 도달 시
    - 유동 참조 | 고정 참조

## 2. 브랜치와 태그 활용하기

### 브랜치 관련 주요 명령어
```bash
git branch # 저장소에 존재하는 브랜치 목록 확인
git branch [브랜치명] # 새 브랜치 생성
git branch -d [브랜치명] # 병합된 브랜치 삭제
git branch -D [브랜치명] # 병합 여부 관계없이 강제 삭제
git switch [브랜치명] # 해당 브랜치로 전환
git switch -c [브랜치명] # 새 브랜치 생성 및 전환
git checkout [브랜치명] # 브랜치 전환 외에도 파일 복원 등 다양한 용도 포함
git checkout -b [브랜치명] # 새 브랜치 생성 및 전환
```

### 병합 처리
```bash
git switch main # 병합을 한 브랜치로 우선 이동
git merge feature/login # 로그인 기능 병합
git merge feature/us    # UI 개선 병합
```
- 충돌 발생
    - 서로 다른 브랜치에서 작업을 하다가 병합을 하게 되면 같은 파일을 건드릴 경우 무조건 충돌이 남
    - 이건 이제 회의해서 처리해야지 어떻게 하는게 좋을 것 같다 등
    - **git status**로 충돌난 파일 확인
    - 어디가 충돌났는지 보여줌
    - 해결 후 스테이징 및 병합

## 주요 버전 관리를 위한 태그 활용하기

### 태그 관리 주요 명령어
```bash
git tag # 현재 저장소에 존재하는 태그 목록 확인
git tag [태그명] # 현재 커밋에 태그 생성
git tag -a [태그명] -m "메시지" # 주석 포함된 태그 생성
git show [태그명] # 특정 태그의 정보 확인
git checkout [태그명] # 해당 태그 시점의 코드로 이동 (detached HEAD)
```

### 시맨틱 버저닝(Semantic Versioning)
- 소프트웨어의 버전을 체계적으로 관리하기 위한 표준
- MAJOR.MINOR.PATCH 형식으로 작성함
    - MAJOR: 이전 버전과 호환되지 않는 큰 변화 (기능 제거, 구조 변경)
    - MINOR: 이전 버전과 호환되는 새로운 기능 추가 (새로운 API 추가)
    - PATCH: 이전 버전과 호환되는 버그 수정 (오류 해결, 최적화)

### 태그 관리 전략
- v1.0.0-alpha: 실험 단계 초기 버전
- v1.0.0-beta: 베타 테스트용
- v1.0.0-rc.1: 출시 후보 버전

## 마무리 정리

### 브랜치 요약
- 브랜치는 작업의 흐름을 분리하고, 충돌을 방지하기 위함
- 병합을 통해 통합하고, 필요 시 삭제하여 정리

### 태그 요약
- 태그는 특정 커밋(시점)을 고정된 이름으로 기록해주는 기능
- 배포 시점, 릴리즈 후보, 긴급 패치 등 중용한 시점을 문서화할 수 있음

### 실전 팁
- 브랜치는 반드시 명확한 목적과 이름을 가지고 만들 것
- 병합 후 사용하지 않는 브랜치는 정리하여 깔끔한 저장소 유지
- 태그는 변경되지 않으므로, 릴리즈 이력을 안전하게 보존 가능
